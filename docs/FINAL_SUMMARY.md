## 💡 关键问题解答

### Q1: LLM 是本地还是云端？

**A**: **混合架构**

- **术语提取**: 本地 jieba（Python）
- **术语评估**: 本地规则（推荐）或 云端 LLM（可选）
- **召回率测试**: 本地（Node.js）
- **权重优化**: 本地 CMA-ES（JavaScript）

**LLM 使用场景**:
- 可选：云端 qwen-turbo 评估（慢，2-5 秒/术语）
- 推荐：本地规则评估（快，<1ms/术语，零成本）

---

### Q2: 每投入新文章就拆分术语吗？

**A**: **批量处理，而非单篇处理**

**推荐策略**:
```
新增文章 → 累积到阈值（如 10 篇） → 批量提取术语 → 更新词典

或者

定期执行（如每周） → 批量处理本周所有新文章 → 更新词典
```

**原因**:
1. 单篇处理效率低（重复运行开销大）
2. 批量处理能更好地统计术语频率
3. 跨文档术语聚合更准确

---

### Q3: 如何建立搜索表？

**A**: **倒排索引 + 术语词典**

**搜索表结构**:
```javascript
{
  "terms": {
    "肺结节": { id: 1, weight: 1.2, domain: "medical_ai" },
    "CT 影像": { id: 2, weight: 1.0, domain: "medical_ai" }
  },
  "inverted_index": {
    1: [doc1, doc3, doc5],  // "肺结节"出现在文档 1,3,5
    2: [doc2, doc4]         // "CT 影像"出现在文档 2,4
  }
}
```

**查询流程**:
```
用户查询："肺结节检测"
   ↓
分词：["肺结节", "检测"]
   ↓
查找倒排索引：[doc1, doc3, doc5] ∩ [doc1, doc5] = [doc1, doc5]
   ↓
返回结果：[doc1, doc5]
```

## 🚀 部署后自进化策略

### 策略 1: 定期批量更新（推荐）

**频率**: 每周执行一次  
**时间**: 每周日凌晨 2 点

**流程**:
```bash
# 每周日凌晨 2 点（cron）
0 2 * * 0 /path/to/weekly_evolution.sh

# 执行：
# 1. 抓取各领域最新文章（5 篇/领域）
# 2. 批量提取术语
# 3. 评估筛选
# 4. 更新词典
# 5. 召回率验证
# 6. 发送报告
```

**优点**:
- 效率高（批量处理）
- 稳定（固定时间执行）
- 可预测（资源消耗可控）

---

### 策略 2: 阈值触发更新

**触发条件**: 新增文章 >= 20 篇

**流程**:
```javascript
// 监控目录
watch('data/articles/', (newFiles) => {
  if (newFiles.length >= 20) {
    runSelfEvolution(); // 立即执行
  }
});
```

**优点**:
- 实时性强
- 响应快速

**缺点**:
- 频繁更新可能影响稳定性
- 资源消耗不可预测

---

### 策略 3: 混合模式（最佳实践）

**定期 + 阈值结合**:
- **定期**: 每周日凌晨 2 点执行
- **阈值**: 新增文章>=20 篇时立即执行

**配置**:
```bash
# crontab
0 2 * * 0 /path/to/weekly_evolution.sh  # 定期任务

# 监控脚本（后台）
nohup node monitor_new_articles.js &  # 阈值触发
```
